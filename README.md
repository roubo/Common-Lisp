Common Lisp再开始
---
因为在和几位Lisp开发者进行交流的时候，当被问及会如何使用common lisp的时候我还是非常的迷惑，毕竟是准备好好学一门对丰富自己编程思想的语言，了解它的优点和应用面应该是最必要的了。
---
在这里需要再次明确的是，在应用面上，Common Lisp与C/C++之类的在适应的平台上的功能作用是一样的，也就是说，在给出了一个需要实现的功能以后，用任何一种语言都应该是有解决方案的。不同的是，哪一种是最适合的，最容易实现的，最高效的，或者甚至是最帅气的。而在决定学习common lisp之前，是因为被告知它可以是最简洁的，最帅气的。
---
还有一个问题就是，common lisp现在还没有给自己呈现出来一个可观的实例，这是最让人发愁的。但是需要认清的是，它肯定是可以做任何能想到的。就像上面说的一样。
---
事实上，一个完整的Lisp实现将带着丰富的运行环境，并提供GUI微件，多线程控制和TCP/IP套接字等。更好地使用lisp，就要关注人们给他编写的各种库。
---
在问及使用那种编译器的时候，其实这是一个概念问题。虽然也有这样的“编译器”将common lisp代码直接编译成可执行文件，但是对lisp代码最好的理解方法应该是让它们纯粹地运行在某个“实现”的“运行时环境”下。那么现在问题就变为我用哪个“实现”，哪个“运行时环境”的问题了。好了，我用的基于SLIME的Lisp in a Box这个“开发环境”。那在这个环境下的代码能有通用性吗？有。因为所有“实现”都是在Common Lisp标准下实现的，所以代码的平台或者“运行时环境”的移植性就能好很多。
---
对于使用Common Lisp的工作环境也是很有意思的，针对Lisp in a Box 来说，在Emacs编辑器下，通过使用一些特有的快捷键，工作便可以在各个缓冲区（文件）之间切换，可以让REPL和lisp代码文件之间很好的配合和切换，甚至是编译。正如所说的，“资深的黑客们经常会保持一个Lisp映像日复一日地运行，不断地添加，重新定义和测试他们的程序。”
